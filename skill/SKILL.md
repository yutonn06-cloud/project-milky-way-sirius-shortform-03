# SNS・ショート動画リライトスキル

原文をもとに、ターゲット・文体・投稿先に最適化されたリライト文を生成する。

## ワークフロー

### 1. 原文の決定

原文の決定方法は以下の優先順位で判断する：

**A. ユーザーが原文を直接提供した場合** → そのまま使う
**B. ユーザーが原文を提供していない場合** → 内蔵の原文集からランダムに1本選ぶ

#### 内蔵原文集の使い方

原文は章別ファイルで `references/` フォルダに格納されている。
合計100本（10章×10本）の原文がある。

スキルのインストール先パスを以下の順で探す：
1. `/mnt/skills/user/sns-rewrite/references/`
2. `/mnt/skills/private/sns-rewrite/references/`

**ファイル構成：**
- `index.md` — 全原文の一覧（まずこれを読む）

セット1・哲学シリーズ（原文001〜100）：
- `ch01_自由と時間.md` 〜 `ch10_知と無知.md`

セット2・スピリチュアルシリーズ（原文101〜250）：
- `s01_ChosenOne.md` 〜 `s15_地球という学校.md`

セット3・社会構造・覚醒シリーズ（原文251〜350）：
- `t01_学校教育の嘘.md` 〜 `t10_分岐する世界と選択.md`

セット4・社会構造・覚醒シリーズ Vol.2（原文351〜450）：
- `v01_学校教育の嘘Vol2.md` 〜 `v10_分岐する世界と選択Vol2.md`

各ファイル内の原文は `===` で区切られ、`## 原文XXX：タイトル` の形式で番号が振られている。

**ランダム選択の手順：**
1. まずランダムにファイルを1つ選ぶ（ch01〜ch10, s01〜s15 の25ファイル）
2. そのファイルを `view` ツールで読み込む
3. ファイル内の10本からランダムに1本選ぶ

**選択ルール：**
- 毎回、異なる原文を選ぶよう意識する（直前に使ったものは避ける）
- ユーザーが「別のを選んで」と言った場合は、必ず前回と異なるものを選ぶ
- ユーザーが「原文03で」のように番号指定した場合は、該当する章ファイルを読み込んでその番号の原文を使う
- 選んだ原文のタイトルと番号をユーザーに伝えてから方針選択に進む

### 2. 方針の選択を提示する

ユーザーが具体的な指示（ターゲット・文体・目的など）を既に明示している場合はそれに従う。
明示していない場合は、`ask_user_input_v0` ツールを使って以下の3つの軸から方針を選んでもらう。

選択肢が多いため、各軸2〜3個に絞って提示する。毎回異なる組み合わせを出すことで、バリエーションが自然に生まれる。

#### 軸A：ターゲット（誰に届ける？）

以下から毎回2〜3個をランダムに選んで提示する：

1. 40代〜50代で、生き方に違和感を感じ始めた女性
2. 「自分の人生これでいいのか」と思い始めた30代〜40代の会社員
3. 周りに合わせて生きてきたけど、もう限界を感じている人
4. スピリチュアルに興味はあるけど、周りには言えずにいる女性
5. 子育てがひと段落して「自分の人生」を取り戻したいと感じている主婦
6. 職場の人間関係に疲れて、ひとりで働く生き方に憧れている人
7. 「自分は普通じゃない」と感じて、ずっと孤独を抱えてきた人
8. 離婚・退職・病気など、人生の転機を経験して価値観が変わった人
9. SNSで自己啓発やスピリチュアルの発信を見て、自分も変わりたいと思い始めた人
10. 組織に属さず、自分の力で生きていきたいと考えている40代〜50代

#### 軸B：文体・トーン

以下から毎回2〜3個をランダムに選んで提示する。**全選択肢とも語尾は敬体（です・ます調）で書くこと。タメ語・カジュアル口調（「ねえ」「でしょ」「だよね」「じゃん」など）は禁止。**

1. やさしく語りかけるけど、核心はズバッと言い切るトーン（敬体維持）
2. 静かだけど力強い、ひとり語りのようなトーン（敬体維持）
3. です・ます調ベースに、体言止めや問いかけを混ぜてリズムを出す
4. 読者の痛みにまず共感してから、視点を反転させる構成（敬体維持）
5. 内省的で詩的、間を活かした語り口（敬体維持）

#### 軸C：投稿先・目的

以下から毎回2〜3個をランダムに選んで提示する：

1. Threads投稿（350〜500文字）
2. X投稿（125〜130文字）
3. ブログ記事本文（600〜700文字）
4. YouTube台本（400〜600文字）
5. Instagram Reelsテロップ（1文10〜15文字）
6. ショート動画スクリプト（300〜400文字）

### 3. リライトの実行（A案・B案の2本生成）

選択された方針に基づいて、**必ず2案（A案・B案）を生成する。**

#### A案とB案の差分ルール
- A案とB案は**同じ原文・同じターゲット・同じ投稿先**で生成する
- **文体・トーン・切り口を変えて差別化する**
- 例：A案＝やさしく語りかけるトーン、B案＝ズバッと言い切るトーン
- 例：A案＝問いかけで始まる構成、B案＝断言で始まる構成
- A案・B案それぞれに異なる冒頭・構成・締めを使い、明確に違いが出るようにする

#### 基本ルール
- **敬体（です・ます調）を必ず使う。A案・B案ともに語尾は「〜です／〜ます／〜でしょうか／〜ません」に統一する。**
- **タメ語・カジュアル表現は禁止**：「ねえ」「でしょ」「だよね」「じゃん」「だよ」「だね」「〜よね」「〜なの？」など、友達口調を想起させる語尾・呼びかけは使わない。
- A案とB案で文体を変える際も、両方とも敬体は維持したまま、構成・切り口・冒頭表現で差別化する。
- 指定された文字数範囲を厳守する
- 原文の核心メッセージを保ちつつ、ターゲットに響く言葉選びに変換する
- 冗長な表現を削り、1文を短くする
- 原文をそのまま短くするのではなく、選んだターゲット・文体に合わせて「書き直す」こと

#### プラットフォーム別の注意点

**Threads：**
- 冒頭1〜2行で「自分のことだ」と思わせる問いかけや断言を入れる
- 締めは余韻を残す。共感や希望で閉じる
- 改行を多めに入れ、視覚的に読みやすくする

**X（Twitter）：**
- 余白を極限まで削ぎ落とす
- 1投稿125〜130文字で完結させる
- インパクトのある1文で始める

**ブログ：**
- 段落ごとに「次を読みたくなる引き」を作る
- 見出しは不要（本文のみ）
- 読者が自分ごとに感じる具体例を織り込む

**YouTube台本：**
- 書き言葉を話し言葉に変換する（「〜なんですよね」「〜じゃないですか」）
- 間（ま）を意識した改行を入れる
- 聞き手が「うんうん」と頷けるリズムにする

**Instagram Reels テロップ：**
- 1文を10〜15文字に抑える
- テンポよく区切る
- 画面映えする短いフレーズ

**ショート動画スクリプト：**
- 最初の3秒で惹きつける一言を入れる
- 60秒以内で読み切れる量にする
- 感情の起伏を意識した構成にする

### 4. 出力フォーマット

リライト結果は以下の形式で2案まとめて出力する：

```
【原文】原文XXX：タイトル
【ターゲット】（選択されたターゲット）
【投稿先】（選択された投稿先）

━━ A案 ━━
【文体】（A案の文体）
【文字数】○○○文字

（A案リライト本文）

━━ B案 ━━
【文体】（B案の文体）
【文字数】○○○文字

（B案リライト本文）
```

### 5. Notionへの書き込み

リライト結果をNotionの「リライトスクリプト」データベースに自動で書き込む。

**書き込み先：**
- データベース「リライトスクリプト」（親ページ「Rewrite for Short Video」配下）
- data_source_id: `e8322351-3390-420a-af36-19d6836bee0c`

**書き込み方法：**
`Notion:notion-create-pages` ツールを使い、データベースに1レコードを追加する。
1レコードにA案・B案の両方を格納する。

**プロパティのマッピング：**

| プロパティ | 値 |
|-----------|-----|
| タイトル | 原文のタイトル（例：「週末が怖い」） |
| リライト本文 | A案の本文テキスト |
| リライト本文B | B案の本文テキスト |
| 投稿先 | 選択肢から1つ（Threads / X / ブログ / YouTube台本 / Reels / ショート動画） |
| ターゲット | 選択されたターゲットの説明文 |
| 文体 | A案の文体＋B案の文体（例：「A: やさしく語りかける / B: ズバッと言い切る」） |
| 原文番号 | 原文XXX |
| 文字数 | A案の文字数（数値） |
| 文字数B | B案の文字数（文字列として書き込む） |
| 採用 | 「未判定」（デフォルト） |
| ステータス | 「未使用」（デフォルト） |
| 音声URL_A | ElevenLabsで生成した音声のURL（Step 6で書き込み） |
| 音声URL_B | ElevenLabsで生成した音声のURL（Step 6で書き込み） |

**呼び出し例：**
```json
{
  "parent": {"data_source_id": "e8322351-3390-420a-af36-19d6836bee0c", "type": "data_source_id"},
  "pages": [{
    "properties": {
      "タイトル": "週末が怖い",
      "リライト本文": "（A案の本文テキスト）",
      "リライト本文B": "（B案の本文テキスト）",
      "投稿先": "Threads",
      "ターゲット": "40代〜50代で、生き方に違和感を感じ始めた女性",
      "文体": "A: やさしく語りかける / B: 共感→反転の構成",
      "原文番号": "原文001",
      "文字数": 420,
      "文字数B": "385",
      "採用": "未判定",
      "ステータス": "未使用"
    }
  }]
}
```

リライト完了後、ユーザーに確認なしで自動的にNotionに書き込む。

### 6. ElevenLabsで音声生成

リライト本文をElevenLabs APIで音声化する。

**前提条件：**
- ユーザーがElevenLabs APIキーを持っていること
- Coworkのタスク実行時は bash_tool でAPIを直接呼び出す

**音声生成の手順：**

A案・B案それぞれについて以下を実行する：

1. リライト本文テキストを用意する
2. ElevenLabs Text-to-Speech APIを呼び出す

```bash
curl -X POST "https://api.elevenlabs.io/v1/text-to-speech/{voice_id}" \
  -H "xi-api-key: {ELEVENLABS_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{
    "text": "（リライト本文）",
    "model_id": "eleven_v3",
    "voice_settings": {"stability": 0.5, "similarity_boost": 0.75}
  }' \
  --output /home/claude/audio_a.mp3
```

3. 生成した音声ファイルを `/mnt/user-data/outputs/` に保存する
4. `present_files` でユーザーに提示する

**注意：**
- ElevenLabs APIキーとvoice_idはユーザーに確認する（初回のみ）
- voice_idが未指定の場合はユーザーに聞く
- ネットワーク制限で api.elevenlabs.io にアクセスできない場合は音声生成をスキップし、その旨を報告する
- 音声ファイルのURLはNotionの音声URL_A・音声URL_Bフィールドには直接書き込めないため、ファイルとして提示する

### 7. バリエーション提案

リライト＆Notion保存の完了後、次のアクションを1つ提案する。例：
- 「同じ原文で、X投稿用の短縮版も作れますがいかがですか？」
- 「別の原文をランダムに選んで、同じ方針でリライトしましょうか？」
- 「ターゲットを変えて、30代会社員向けのバージョンも作れます。」

こうすることで、テンポよく複数のコンテンツを量産できる。
